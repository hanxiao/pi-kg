=================  WHAT "DONE" LOOKS LIKE (optimize for these outcomes)  =================

1. FORMAT. Each line of KG.jsonl is one edge as compact JSON:
   {"subject":"<canonical entity>","predicate":"<snake_case relation, <=32 chars>","object":"<canonical entity OR short atomic value>","title":"<=140 chars>","description":"<=350 chars, carries the answer + evidence>","evidence_span":"<verbatim substring of a source doc>","source_files":["<doc filename>", ...],"confidence":<0-100>}
   The whole file must be valid JSONL (every line parses).

2. GROUNDED. Every fact is supported by the corpus: evidence_span is a verbatim substring of the source document it came from. No invented facts, no facts that are not in the documents. Favor specific facts (proper nouns, versions, numbers, dates, metric scores) over generic claims. Skip boilerplate/nav/login/error pages.

3. CANONICAL, DEDUPLICATED NODES (the property that matters most). The same real-world entity must be ONE node - one identical string - everywhere it appears, across all documents. The final graph must have:
   - NO duplicate entities: surface variants of the same thing ("the Jina AI team", "Jina AI", "Jina") collapse to a single canonical node; "jina-embeddings-v3" and "a model called jina-embeddings-v3" are the same node.
   - NO wrong merges: genuinely different entities stay distinct (jina-embeddings-v2 != v3; a base model != its Chinese variant; different versions/dates/values are different nodes). Different TYPES never merge even when strings overlap: an organization (Jina AI), a product/service (Jina Reader), a URL/API endpoint/domain (r.jina.ai), a model, a paper, and a person are all separate nodes. A concrete WRONG merge to avoid: r.jina.ai (endpoint) -> Jina AI (company). Embedding similarity OVER-proposes such merges; decide using each string's usage context (its predicates/neighbors), not the raw string - matching context means same entity, differing context (api_endpoint vs has_model) means keep separate.
   - subject and object are always canonical ENTITIES or short atomic VALUES (a name, date, number+unit, version, score) - never a sentence or clause.
   - predicates are precise snake_case relations, and synonymous relations are unified to one canonical predicate (no two predicates meaning the same thing); avoid vague catch-alls like related_to / affiliated_with.

4. NO DUPLICATE EDGES. After canonicalization, there is at most one edge per (subject, predicate, object); facts asserted by multiple documents are merged into one edge whose source_files lists all of them.

5. WELL-CONNECTED. Because the graph will be used to find long multi-hop chains, it must actually connect: a healthy fraction of entities appear in >=2 documents and the largest connected component covers a large share of the nodes. A fragmented graph (mostly singleton nodes) means canonicalization/deduplication was insufficient - keep improving until it is well-connected.

Also write KG_STATS.md reporting: total edges, total canonical nodes, number of entities appearing in >=2 documents, number of connected components, and largest-connected-component size. KG.jsonl is the only deliverable that is graded; make sure it exists and meets bars 1-5 before you stop.

CONSTRAINT - context window is only ~56k tokens: do not load many whole documents into context at once (that overflows and wastes the turn). Persist progress to KG.jsonl as you go so it survives context compaction between turns. How exactly you sequence extraction, deduplication, and verification is up to you - use your embedding tools where they help (e.g. to find which surface strings are the same entity), but the embeddings only propose; you decide the final merges. Iterate until all five outcomes above hold.
