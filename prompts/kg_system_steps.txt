=================  OUTPUT FORMAT  =================
Each line of KG.jsonl is exactly one edge (one fact), a compact JSON object:
{"subject":"<canonical entity>","predicate":"<snake_case relation, <=32 chars>","object":"<canonical entity OR short atomic value>","title":"<=140 char natural sentence stating the fact","description":"<=350 chars: the answer + evidence, may include a short verbatim quote","evidence_span":"<verbatim substring of a source doc>","source_files":["<doc filename>"],"confidence":<0-100>}

=================  THE CANONICAL-NODE RULE (most important)  =================
subject and object MUST be canonical ENTITIES or short atomic VALUES, never prose. They are graph nodes: the same entity must come out byte-identical every time so edges connect across documents.
- Shortest canonical name of a real entity: a person, org, product/model, dataset, method, paper, place, technology, or a concrete atomic value (date, number+unit, version, metric score).
- Strip articles/roles/qualifiers: "the Jina AI team" -> "Jina AI"; "a model called jina-embeddings-v3" -> "jina-embeddings-v3".
- Reuse the EXACT same string for the same entity across every fact. This is how nodes merge.
- Keep genuinely different entities distinct: jina-embeddings-v2 != v3; a base model != its Chinese variant.
- TYPE DISTINCTION (do NOT merge across these types even when strings look similar): an organization/company (Jina AI), a product/service (Jina Reader), a URL/API endpoint/domain (r.jina.ai, api.jina.ai), a model (jina-embeddings-v3), a paper, and a person are ALL different node types and must stay separate nodes. Example of a WRONG merge to avoid: r.jina.ai (an endpoint) -> Jina AI (a company). A service's endpoint/URL is its own node (or a value), never the company that owns it.
- Never put a sentence/clause in subject/object. If a value is descriptive, make object the atomic value and explain in description.
- Prefer entity-entity edges (they make the graph rich); entity-value edges are fine too.
PREDICATES: pick the MOST precise snake_case relation; coin new ones freely; AVOID vague catch-alls (related_to, affiliated_with). Style examples only (not a menu): built_by, based_on, trained_on, released_on, outperforms, evaluated_on, scored, integrates_with, acquired_by, authored_by, cites, successor_of, has_context_length, has_dimension.
