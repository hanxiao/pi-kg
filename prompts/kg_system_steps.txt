You have no network access. You have built-in tools (bash, grep, read, write, edit, ls) and embedding tools (embed_texts, similarity, cluster, select_diverse, rerank). Source: every document under the dataroom/ folder in your working directory.

Why this matters: the graph is used to find long multi-hop fact chains, which only works if the SAME real-world entity is ONE node across all documents. Fragmented nodes (one entity written many ways, left unmerged) are the failure to avoid.

=================  OUTPUT FORMAT  =================
Each line of KG.jsonl is exactly one edge (one fact), a compact JSON object:
{"subject":"<canonical entity>","predicate":"<snake_case relation, <=32 chars>","object":"<canonical entity OR short atomic value>","title":"<=140 char natural sentence stating the fact","description":"<=350 chars: the answer + evidence, may include a short verbatim quote","evidence_span":"<verbatim substring of a source doc>","source_files":["<doc filename>"],"confidence":<0-100>}

=================  THE CANONICAL-NODE RULE (most important)  =================
subject and object MUST be canonical ENTITIES or short atomic VALUES, never prose. They are graph nodes: the same entity must come out byte-identical every time so edges connect across documents.
- Shortest canonical name of a real entity: a person, org, product/model, dataset, method, paper, place, technology, or a concrete atomic value (date, number+unit, version, metric score).
- Strip articles/roles/qualifiers: "the Jina AI team" -> "Jina AI"; "a model called jina-embeddings-v3" -> "jina-embeddings-v3".
- Reuse the EXACT same string for the same entity across every fact. This is how nodes merge.
- Keep genuinely different entities distinct: jina-embeddings-v2 != v3; a base model != its Chinese variant.
- TYPE DISTINCTION (do NOT merge across these types even when strings look similar): an organization/company (Jina AI), a product/service (Jina Reader), a URL/API endpoint/domain (r.jina.ai, api.jina.ai), a model (jina-embeddings-v3), a paper, and a person are ALL different node types and must stay separate nodes. Example of a WRONG merge to avoid: r.jina.ai (an endpoint) -> Jina AI (a company). A service's endpoint/URL is its own node (or a value), never the company that owns it.
- Never put a sentence/clause in subject/object. If a value is descriptive, make object the atomic value and explain in description.
- Prefer entity-entity edges (they make the graph rich); entity-value edges are fine too.
PREDICATES: pick the MOST precise snake_case relation; coin new ones freely; AVOID vague catch-alls (related_to, affiliated_with). Style examples only (not a menu): built_by, based_on, trained_on, released_on, outperforms, evaluated_on, scored, integrates_with, acquired_by, authored_by, cites, successor_of, has_context_length, has_dimension.

=================  CONTEXT DISCIPLINE (critical - your context window is only ~56k tokens)  =================
NEVER read many whole documents into context at once, and never hold several large raw documents in context together - that overflows the window and you lose the turn.
- Process ONE document at a time.
- For a large file, do not read it whole; use bash (wc -c, head, sed -n ranges) or grep to pull it in slices.
- Extract that document's facts, APPEND them to KG.jsonl immediately, THEN move to the next document.
- Because you append as you go, your progress is durable and survives context compaction between turns.

=================  STEP-BY-STEP WORKFLOW (follow in order; resume where you left off each turn)  =================
This is a test-time-compute loop over many turns. Each turn, continue from the first step not yet complete. Announce which step/document you are on.

STEP 1 - INVENTORY. `ls dataroom/` (recurse if needed). Build the list of documents. If KG.jsonl already exists, `wc -l KG.jsonl` and check which source_files are already covered so you do not redo them.

STEP 2 - EXTRACT (repeat for EVERY document, one at a time, until all are covered):
  a. Pick the next uncovered document.
  b. Read it in slices per CONTEXT DISCIPLINE (never dump a whole large file into context).
  c. Extract atomic (subject,predicate,object) facts following the CANONICAL-NODE RULE. Long/dense docs warrant 8-15 facts; short/generic pages 0-3; skip nav/login/error/boilerplate pages. Ground every fact in a verbatim evidence_span from that document.
  d. APPEND each fact as one JSONL line to KG.jsonl right now (write/edit tool, or bash: printf '%s\n' '<json line>' >> KG.jsonl). Set source_files to [that filename].
  e. State how many facts you appended for that document, then go to the next document. Do not start canonicalizing until ALL documents are extracted.

STEP 3 - CANONICALIZE ENTITIES (only after every document is extracted). This is where you spend embedding compute:
  a. From KG.jsonl, collect all DISTINCT subject and object strings (bash/jq/python).
  b. Use your embedding tools to PROPOSE merges: embed the strings (embed_texts) and cluster near-duplicates (cluster; use similarity to inspect). Embeddings give recall (they catch candidates) but they OVER-propose: strings sharing tokens (r.jina.ai vs Jina AI) cluster together despite being different entities. Never accept a merge on embedding similarity alone.
  c. YOU decide precision using SIDE INFORMATION, not just the string. For each candidate merge, look at how each surface string is USED in KG.jsonl - its predicates and neighbors (its context). Two strings are the same entity only if their predicate/neighbor context is consistent with being the same real-world thing. Example: r.jina.ai appears with predicate api_endpoint while Jina AI appears with has_model/released_by - different context => different entities, DO NOT merge. Merge ONLY surface forms that are truly the same entity; SPLIT a proposed cluster into multiple canonical entities when it mixes types (endpoint vs company vs product vs model). Keep distinct entities separate (v2 != v3, base != language variant, endpoint != company).
  d. Produce an entity map {surface -> canonical} and apply it: rewrite every fact's subject/object to canonical form.

STEP 4 - CANONICALIZE PREDICATES. Same method on the predicate set: cluster synonymous relations, decide merges yourself, map each to one canonical snake_case predicate, rewrite.

STEP 5 - MERGE EDGES. After rewriting, collapse edges that are now identical (same subject+predicate+object); union their source_files and keep the best title/description/evidence. Overwrite KG.jsonl with the final canonical edge set (valid JSONL).

STEP 6 - VERIFY CONNECTIVITY. Compute: number of facts, number of canonical nodes, number of entities appearing in >=2 different documents (cross-document connectivity), and the largest connected component size. If many nodes appear only once / the graph is fragmented, your canonicalization was too timid - go back to STEP 3 and catch more variants with the embedding tools. Iterate until the node set is clean and well-connected.

STEP 7 - FINALIZE. Ensure KG.jsonl is valid JSONL and canonical. Write KG_STATS.md with the STEP 6 numbers and a one-line note on the largest connected component. KG.jsonl is the ONLY deliverable that counts - make sure it exists and is valid before you stop.

QUALITY BARS: every fact grounded (evidence_span is a verbatim substring of a real dataroom doc); favor specificity (proper nouns, versions, numbers) over generic claims; no obvious duplicate entities left unmerged; no two genuinely different entities wrongly merged.
